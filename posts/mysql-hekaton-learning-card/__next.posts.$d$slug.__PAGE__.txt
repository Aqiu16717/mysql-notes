f:T11e4,<h1 id="一张图看懂hekaton--唯一量产的无锁-oltp-引擎">一张图看懂：Hekaton — 唯一量产的无锁 OLTP 引擎</h1>
<blockquote>
<p><strong>一句话：</strong> SQL Server 2014 内置的 Hekaton 证明了无锁可以量产——但代价是"功能换性能"。它是 MySQL 最值得研究的工程教训。</p>
</blockquote>
<hr>
<h2 id="为什么-hekaton-特别">为什么 Hekaton 特别？</h2>
<pre><code class="language-mermaid">graph LR
    A[Shore-MT&#x3C;br/>学术基线] --> B[Silo&#x3C;br/>研究原型]
    B --> C[Hekaton&#x3C;br/>SQL Server 2014 GA]
    C --> D[服务数百万用户]
    
    style C fill:#4caf50,color:white
    style D fill:#4caf50,color:white
</code></pre>
<p><strong>4 篇论文中唯一一个真正出货的系统。</strong> 它的生产故障模式比任何学术数据更有价值。</p>
<hr>
<h2 id="无锁如何工作">无锁如何工作？</h2>
<pre><code class="language-mermaid">graph TD
    TXN[事务] -->|读| MVCC[行版本链&#x3C;br/>begin_ts ~ end_ts]
    TXN -->|写| NEW[插入新版本&#x3C;br/>begin = 全局时间戳]
    TXN -->|提交| VALIDATE{乐观验证&#x3C;br/>读集是否被改过?}
    
    VALIDATE -->|通过| COMMIT[分配 commit ts&#x3C;br/>写入日志]
    VALIDATE -->|冲突| RETRY[abort + 重试]
    
    COMMIT --> GC[后台 GC&#x3C;br/>清理旧版本]
    
    style VALIDATE fill:#ff9800,color:white
    style RETRY fill:#f44336,color:white
</code></pre>
<p><strong>关键差异：</strong> 无锁 = 无等待。冲突不阻塞，直接 abort + retry。</p>
<hr>
<h2 id="-三个量产失败模式">🔴 三个量产失败模式</h2>
<table>
<thead>
<tr>
<th>故障</th>
<th>表现</th>
<th>根因</th>
<th>MySQL 对等物</th>
</tr>
</thead>
<tbody>
<tr>
<td><strong>GC 版本清理滞后</strong></td>
<td>10K 版本链 → 40-60% 吞吐量下降</td>
<td>后台清理跟不上写入速度</td>
<td>InnoDB purge thread (<code>row0purge.cc</code>)</td>
</tr>
<tr>
<td><strong>验证 abort 风暴</strong></td>
<td>热点行 90%+ abort 率 → CPU 空转在 retry</td>
<td>乐观并发 + 热点 = 重试死循环</td>
<td><code>innodb_spin_wait_delay</code> 高竞争抖动</td>
</tr>
<tr>
<td><strong>检查点碎片</strong></td>
<td>恢复时间 ∝ 碎片数量</td>
<td>checkpoint 文件无限增长</td>
<td>类似 InnoDB redo log 无限增长场景</td>
</tr>
</tbody>
</table>

<h2 id="修复了什么">修复了什么？</h2>
<table>
<thead>
<tr>
<th>故障</th>
<th>Hekaton 修复（SQL Server 2016）</th>
<th>对 MySQL 的启示</th>
</tr>
</thead>
<tbody>
<tr>
<td>GC 滞后</td>
<td>① 版本链硬上限（默认 100）→ 超过后<strong>阻塞写</strong> ② 合作式 GC：每个读线程顺手清理 1~2 个版本</td>
<td>给 <code>row0purge.cc</code> 加一个"读路径顺手清理"的 hook，上限 2 版本/读</td>
</tr>
<tr>
<td>Abort 风暴</td>
<td>① 指数退避 + 抖动（1ms→10ms→100ms）② 热点行哈希分桶</td>
<td><code>innodb_spin_wait_delay</code> 升级到事务级退避</td>
</tr>
<tr>
<td>检查点碎片</td>
<td>增量检查点 + 后台合并</td>
<td>InnoDB 已有增量 checkpoint，差距在碎片合并</td>
</tr>
</tbody>
</table>

<h2 id="mysql-能不能用">MySQL 能不能用？</h2>
<table>
<thead>
<tr>
<th></th>
<th>判断</th>
</tr>
</thead>
<tbody>
<tr>
<td>❌ <strong>不能整体搬</strong></td>
<td>InnoDB 的 undo-log MVCC ≠ Hekaton 的行内版本存储——需要全新 <code>ha_hekaton</code> handler</td>
</tr>
<tr>
<td>❌ <strong>最大 blocker</strong></td>
<td>Hekaton 只有 SI + Serializable，<strong>无 READ COMMITTED</strong>——MySQL 生态离不开 RC</td>
</tr>
<tr>
<td>✅ <strong>可以学</strong></td>
<td>合作式 GC（读路径清理 1-2 版本）、事务级指数退避、版本链硬上限</td>
</tr>
</tbody>
</table>
<hr>
<h2 id="记住这三件事">记住这三件事</h2>
<ol>
<li><strong>无锁的代价是功能面缩小</strong> — Hekaton 放弃了 FK、DML 触发器、READ COMMITTED，不是免费的午餐</li>
<li><strong>生产故障比实验室性能数据重要 10 倍</strong> — GC 滞后、abort 风暴、检查点碎片才是真实问题</li>
<li><strong>合作式 GC 是最立刻可用的启示</strong> — 不需要任何锁免基础设施，只是一个读路径 hook</li>
</ol>
