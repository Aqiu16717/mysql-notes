f:T1239,<h1 id="一张图看懂bwtree--为什么-ahi-是-mysql-的无锁-b-tree-起点">一张图看懂：BwTree — 为什么 AHI 是 MySQL 的无锁 B-tree 起点</h1>
<blockquote>
<p><strong>一句话：</strong> BwTree 用映射表 + CAS 消除了 B-tree 所有锁。惊喜发现：InnoDB 的 AHI（自适应哈希索引）已经是 BwTree 的思想雏形。</p>
</blockquote>
<hr>
<h2 id="传统-b-tree-的锁痛点">传统 B-tree 的锁痛点</h2>
<pre><code class="language-mermaid">graph TD
    ROOT[根页&#x3C;br/>⚠️ 每次访问都要锁] --> L1[中间页&#x3C;br/>⚠️ 分裂时排他锁传播]
    L1 --> L2[中间页]
    L2 --> LEAF[叶页]
    
    SPLIT[页分裂] -->|需要 X-lock| ROOT
    SPLIT -->|X-lock 逐层向上| L1
    
    style ROOT fill:#f44336,color:white
    style L1 fill:#ff9800,color:white
</code></pre>
<ul>
<li><strong>根页</strong> = 热点：每个事务都要访问，锁竞争最激烈</li>
<li><strong>页分裂</strong> = 长排他锁：从叶到根逐层 X-lock，阻塞整棵子树</li>
</ul>
<hr>
<h2 id="bwtree-怎么消除锁">BwTree 怎么消除锁？</h2>
<pre><code class="language-mermaid">graph TD
    PID[逻辑 Page ID] -->|查找| MT[映射表&#x3C;br/>PID → 物理页指针]
    MT -->|CAS 原子切换| OLD[旧页]
    MT -->|CAS 原子切换| NEW[新页]
    
    SPLIT2[页分裂] -->|CAS 更新映射表&#x3C;br/>无锁!| MT
    READ[读请求] -->|始终看到&#x3C;br/>且新或旧的一致视图| MT
    
    style MT fill:#4caf50,color:white
    style SPLIT2 fill:#2196f3,color:white
</code></pre>
<p><strong>关键思想：不直接指向页，通过映射表间接访问。</strong> 分裂 = CAS 更新映射表，读者看到旧映射或新映射——都是正确的。</p>
<hr>
<h2 id="-ahi--mysql-的-bwtree-雏形">🎯 AHI = MySQL 的 BwTree 雏形</h2>
<p><strong>InnoDB 的 Adaptive Hash Index (<code>btr0sea.cc</code>)</strong>:</p>
<pre><code>AHI 做的事：                        BwTree 需要做的事：
(索引ID, 键前缀) → 页指针           (Page ID) → 物理页指针
   ↑ 已经是无锁的！                    ↑ 从这里扩展！
</code></pre>
<p><strong>AHI 已经维护了一个无锁的 key → page 映射！</strong> 把它从"键前缀哈希 → 页"推广为"页 ID → 物理页"——就是从 AHI 到 BwTree 映射表的距离。</p>
<hr>
<h2 id="mysql-能不能用">MySQL 能不能用？</h2>
<table>
<thead>
<tr>
<th></th>
<th>判断</th>
<th>风险</th>
</tr>
</thead>
<tbody>
<tr>
<td>✅ <strong>CAS 页分裂</strong></td>
<td>只改分裂路径，不动读路径。替换 <code>btr_page_split_and_insert</code> 中的 X-lock 为 CAS 映射表更新</td>
<td>低——无新锁序边，可证明无死锁</td>
</tr>
<tr>
<td>⚠️ <strong>AHI → 完整映射表</strong></td>
<td>AHI 是自适应的（懒构建、压力下驱逐），映射表必须完整——从"缓存热门"到"索引一切"的跃迁</td>
<td>中——需独立缓冲区分配，小 buf_pool 不适用</td>
</tr>
<tr>
<td>❌ <strong>全 BwTree（含 delta 链）</strong></td>
<td>Delta 链在内存快（~5ns），在磁盘灾难（~5ms/page）。InnoDB 是 disk-based</td>
<td>高——仅适用于内存优化表</td>
</tr>
</tbody>
</table>
<hr>
<h2 id="️-最务实的渐进路径">🛤️ 最务实的渐进路径</h2>
<pre><code class="language-mermaid">graph LR
    NOW[现在&#x3C;br/>InnoDB B+tree&#x3C;br/>rw_lock] -->|第1步| CAS[CAS 页分裂&#x3C;br/>保留读锁&#x3C;br/>只改写路径]
    CAS -->|第2步| AHI2[AHI 推广&#x3C;br/>→ 完整映射表&#x3C;br/>buf_pool > X GB 时启用]
    AHI2 -->|第3步| READ[可选：无锁读&#x3C;br/>仅限内存优化表]
    
    style CAS fill:#4caf50,color:white
    style NOW fill:#2196f3,color:white
</code></pre>
<p><strong>第 1 步就是最大的胜利</strong> —— 页分裂是最长排他锁的源头，用 CAS 消灭它，不改任何读路径。</p>
<hr>
<h2 id="记住这三件事">记住这三件事</h2>
<ol>
<li><strong>AHI 已经是 BwTree 的思想雏形</strong> — 无锁 key→page 映射就在 <code>btr0sea.cc</code> 里，不是从零开始</li>
<li><strong>CAS 页分裂是渐进式改进的黄金路径</strong> — 可证明无死锁，不改读路径，先解决最痛的点</li>
<li><strong>Delta 链在磁盘上是灾难</strong> — 全 BwTree 只能用于内存表，InnoDB 主力场景是 disk-based</li>
</ol>
