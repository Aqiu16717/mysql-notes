22:T2ccf,<h1 id="silososp-2013">Silo（SOSP 2013）</h1>
<h2 id="多核内存数据库的高速事务处理">多核内存数据库的高速事务处理</h2>
<p><strong>作者：</strong> Stephen Tu、Wenting Zheng、Eddie Kohler、Barbara Liskov、Samuel Madden（MIT CSAIL）</p>
<hr>
<h2 id="1-背景">1. 背景</h2>
<p>到 2013 年，多核硬件上的 OLTP 引擎已碰壁：当核心数超过 8-16 时，锁争用和闩锁开销消耗了 14-28% 的 CPU 周期（Harizopoulos 等人，SIGMOD 2008）。主流方案——基于两阶段锁（2PL）的细粒度行级锁——在争用下产生了死锁检测开销、锁表争用和上下文切换风暴。</p>
<p>Silo 提出了一个问题：<em>我们能否为多核时代构建一个完全可序列化的、无锁的 OLTP 引擎？</em></p>
<hr>
<h2 id="2-问题">2. 问题</h2>
<p>传统 OLTP 引擎将 16-30% 的执行时间消耗在：</p>
<ul>
<li><strong>锁管理器争用</strong>（锁表哈希冲突、死锁检测图遍历）</li>
<li><strong>B+树页闩锁争用</strong>（即使使用闩锁耦合，从根到叶的锁定也会阻塞读操作）</li>
<li><strong>日志缓冲争用</strong>（单个全局日志互斥锁串行化提交）</li>
<li><strong>事务协调开销</strong>（2PL 锁获取/释放、两阶段提交）</li>
</ul>
<p>问题本质是<strong>架构性的</strong>：这些机制假设内存稀缺且核心数少。在拥有充足 RAM 和 32+ 核心的现代硬件上，争用成本压倒了有效工作。</p>
<hr>
<h2 id="3-方案">3. 方案</h2>
<p>Silo 的设计建立在三个支柱上：</p>
<h3 id="31-基于-epoch-的组提交">3.1 基于 Epoch 的组提交</h3>
<ul>
<li>时间被划分为<strong>epoch</strong>（周期性递增的全局计数器）</li>
<li>Epoch E 中的事务在 epoch E-1 中所有事务提交<strong>之后</strong>才提交</li>
<li>一个专用的<strong>epoch 线程</strong>推进 epoch 并触发组提交</li>
<li>一个 epoch 中的所有写入被批量处理为一次串行提交批次</li>
<li>这完全消除了日志互斥锁——只有 epoch 线程接触日志</li>
</ul>
<h3 id="32-确定性序列化">3.2 确定性序列化</h3>
<ul>
<li>Silo 不为每行加锁，而是根据读/写集合为每个事务分配<strong>事务 ID（TID）</strong></li>
<li>事务在提交时按 TID 顺序序列化</li>
<li>读集验证：提交时，每次读取的 TID 必须与其读取时的版本匹配</li>
<li>写集应用：所有写入在 epoch 边界原子性地应用</li>
<li>无死锁（无锁获取顺序），无锁表</li>
</ul>
<h3 id="33-masstree无锁-b树">3.3 Masstree（无锁 B+树）</h3>
<ul>
<li>使用 Masstree（一种缓存友好的 trie+B+树混合结构）作为索引</li>
<li>Masstree 通过带版本号的叶节点提供无锁点读</li>
<li>结构修改（分裂/合并）在内节点上使用 CAS</li>
</ul>
<h3 id="关键创新基于-epoch-的内存回收ebr">关键创新：基于 Epoch 的内存回收（EBR）</h3>
<ul>
<li>每个线程公布其<strong>当前 epoch</strong></li>
<li>垃圾回收时：在 epoch E 中释放的内存只有在<strong>所有线程</strong>都已推进到超过 epoch E 后才能回收</li>
<li>快速路径上无引用计数，无 hazard pointer</li>
</ul>
<hr>
<h2 id="4-权衡">4. 权衡</h2>
<table>
<thead>
<tr>
<th>收益</th>
<th>代价</th>
</tr>
</thead>
<tbody>
<tr>
<td>接近线性的 32 核扩展（700K TPC-C TPS）</td>
<td><strong>Epoch 延迟死亡螺旋</strong>：一个慢线程（如长时间运行的 <code>SELECT</code>）阻塞所有内存回收 → OOM</td>
</tr>
<tr>
<td>零死锁检测开销</td>
<td>全或无序列化：epoch 中任一事务冲突则整个 epoch 批次中止</td>
</tr>
<tr>
<td>无锁表，热路径上无闩锁</td>
<td>仅内存——原始设计中无基于磁盘的恢复路径</td>
</tr>
<tr>
<td>简化的正确性验证：基于 TID 的序列化图可轻松验证</td>
<td>不能容忍线程崩溃——持有过期 epoch 的死线程会永久泄漏内存</td>
</tr>
<tr>
<td>确定性：相同输入 → 相同序列顺序（调试友好）</td>
<td>组提交延迟底限 = epoch 间隔（取决于配置，约 1-40ms）</td>
</tr>
</tbody>
</table>
<hr>
<h2 id="5-论文摘要中未提及的失败模式">5. 论文摘要中未提及的失败模式</h2>
<p><strong>来自后续工作（ERMIA，SIGMOD 2016；Foedus，SIGMOD 2014）：</strong></p>
<ol>
<li>
<p><strong>Epoch 停顿级联效应</strong>：一个扫描 1000 万行的只读事务阻止所有其他线程的内存回收。在 Silo 的 TPC-C 工作负载中（短事务）这个问题被掩盖了，但在生产规模下，任何分析查询都会在几秒内导致 OOM。</p>
</li>
<li>
<p><strong>中止放大</strong>：在高争用工作负载下（如 50 个线程争用 TPC-C 中同一个仓库），一个冲突事务会导致整个 epoch 批次中止。热点争用下吞吐量崩溃 60-80%。</p>
</li>
<li>
<p><strong>Masstree 分裂风暴</strong>：在插入密集型工作负载下，Masstree 分裂引发级联 epoch 停顿，因为等待提交的线程必须等待分裂完成的线程推进 epoch。</p>
</li>
<li>
<p><strong>NUMA 不敏感</strong>：Silo 的原始评估在单插槽 32 核机器上进行。在多插槽 NUMA 上，基于 TID 的序列化产生跨插槽一致性流量，扩展性差（Kim 等人，VLDB 2015，双插槽比单插槽吞吐量损失 50%）。</p>
</li>
</ol>

<h3 id="mysql-特有障碍">MySQL 特有障碍：</h3>
<ul>
<li><strong>mem_root 生命周期</strong>：Silo 的 EBR 假定在 epoch 推进时调用 <code>free()</code>。MySQL 的 <code>MEM_ROOT</code> 使用区域语义——不能释放单个对象。需要感知 epoch 的区域或每个 epoch 一个 <code>MEM_ROOT</code>。</li>
<li><strong>KILL CONNECTION</strong>：Silo 没有等价于 MySQL KILL 的机制。中止 epoch 中期的事务需要 epoch 范围中止或延迟 kill（标志 + 惰性检查）。</li>
<li><strong>Binlog 集成</strong>：Silo 的确定性序列顺序不能干净地映射到 binlog 的语句/行格式。epoch 提交批次需要一个能保留顺序的 binlog 事件类型。</li>
</ul>

<h2 id="8-性能论断附实验条件">8. 性能论断（附实验条件）</h2>
<p>**声称：**32 核上 700K TPC-C 事务/秒。"接近线性扩展"。</p>
<p><strong>实验条件：</strong></p>
<ul>
<li>**硬件：**单插槽 32 核 Intel Xeon E5-4620（2.2 GHz），128 GB RAM</li>
<li>**数据集：**TPC-C，100 仓库（全内存）</li>
<li>**工作负载：**100% 内存，无磁盘 I/O，短事务（平均 &#x3C; 10 读，&#x3C; 5 写）</li>
<li>**对比对象：**Shore-MT 基线（传统基于锁的 OLTP），同硬件约 280K TPS</li>
<li>**扩展声明：**1 核 → 32 核 = 24.7 倍加速（77% 并行效率）</li>
</ul>
<p><strong>注意事项：</strong></p>
<ul>
<li>仅单插槽——无 NUMA 评估</li>
<li>无分析查询（纯 OLTP）</li>
<li>未评估崩溃恢复路径</li>
<li>Epoch 间隔调至 40ms（在吞吐量指标中隐藏批处理延迟）</li>
<li>这是 Shore-MT 的 2.5 倍，不是 10 倍——对消除锁开销来说是现实的数字</li>
</ul>
<hr>
<h2 id="9-mysql-生态启示">9. MySQL 生态启示</h2>
<ol>
<li>
<p>**Epoch 模型与 MySQL 的连接模型根本不相容。**MySQL 支持任意时长的连接（如 <code>SELECT SLEEP(3600)</code>），会阻塞基于 epoch 的内存回收。Silo 对短事务（纯 OLTP）的假设对真实 MySQL 部署不成立。</p>
</li>
<li>
<p><strong>确定性序列化完全移除了 InnoDB 的锁系统</strong>——但 MySQL 的 <code>performance_schema</code> 和 <code>sys</code> schema 暴露了监控工具所依赖的锁等待数据。迁移需要替代的监控原语。</p>
</li>
<li>
<p>**基于 TID 的序列化可能映射到组复制（MGR）。**MGR 的基于认证的冲突检测（Galera write-set replication）已经做了类似的工作：跨节点序列化事务。Silo 的确定性 TID 分配在原则上可以与 MGR 的认证集成，产生一个完全确定性的集群。这是最有趣的架构路径，在 MySQL 中尚未被探索。</p>
</li>
</ol>
