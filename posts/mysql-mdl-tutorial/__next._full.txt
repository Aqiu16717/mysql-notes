22:T604b,<h1 id="mysql-mdl-入门到精通metadata-lock-的获取等待与死锁">MySQL MDL 入门到精通：Metadata Lock 的获取、等待与死锁</h1>

<p><strong>一句话理解 MDL：</strong> MDL（Metadata Lock）是 MySQL 的"表结构锁"——它确保你在读数据（SELECT）时，别人不能同时改表结构（ALTER TABLE）。它是 DDL 和 DML 之间的仲裁者。</p>

<h2 id="1-为什么需要-mdl">1. 为什么需要 MDL？</h2>
<p>想象这个场景：</p>
<pre><code>时间线  线程 A (SELECT)              线程 B (ALTER TABLE)
─────────────────────────────────────────────────────────
T1      打开表 t1，开始读数据...
T2                                   删除列 col_3
T3      读到第 100 行                 ALTER 完成
T4      尝试读 col_3 → 💥 崩溃！
</code></pre>
<p>线程 A 打开表时 t1 有 10 列，读到一半时线程 B 删除了 col_3——线程 A 的表结构视图和磁盘上的实际结构不一致了。</p>
<p><strong>MDL 做的事：</strong> 在 T1 给线程 A 发一把"共享读锁"（SR），在 T2 线程 B 申请"排他锁"（X）时被阻塞——直到线程 A 释放 SR。这样线程 A 的整个读过程看到的是同一个表结构。</p>

<h3 id="21-兼容矩阵">2.1 兼容矩阵</h3>
<pre><code>            IX  S  SH  SR  SW SWLP SU SRO SNW SNRW X
IX          +   +   +   +   +   +   +   +   +   +   -
S           +   +   +   +   +   +   +   +   +   -   -
SH          +   +   +   +   +   +   +   +   +   -   -
SR          +   +   +   +   +   +   -   -   -   -   -
SW          +   +   +   +   +   +   -   -   -   -   -
SU          +   +   +   -   -   -   -   -   -   -   -
SRO         +   +   +   -   -   -   -   -   -   -   -
SNW         +   +   +   -   -   -   -   -   -   -   -
SNRW        +   -   -   -   -   -   -   -   -   -   -
X           -   -   -   -   -   -   -   -   -   -   -
</code></pre>
<p><code>+</code> = 兼容（可同时持有），<code>-</code> = 冲突（后来的必须等待）</p>

<h2 id="4-fast-path为什么-mdl-不拖慢你的查询">4. Fast Path：为什么 MDL 不拖慢你的查询</h2>
<p><strong>这是 MDL 最重要的性能设计。</strong> 绝大多数 DML 操作（99%+）的 MDL 请求走 fast path——零链表操作，零等待队列检查，零死锁检测。</p>
<h3 id="41-哪些锁走-fast-path">4.1 哪些锁走 fast path？</h3>
<p>四种轻量锁：</p>
<ul>
<li><code>MDL_SHARED</code> (S)</li>
<li><code>MDL_SHARED_HIGH_PRIO</code> (SH)</li>
<li><code>MDL_SHARED_READ</code> (SR) ← SELECT 就用这个</li>
<li><code>MDL_SHARED_WRITE</code> (SW) ← INSERT/UPDATE/DELETE 用这个</li>
</ul>
<h3 id="42-fast-path-怎么工作">4.2 Fast Path 怎么工作？</h3>
<pre><code>正常路径（slow path）:                Fast Path:
─────────────────────                ─────────
创建 ticket                          只 inc 原子计数器
挂入 MDL_lock.granted 链表           不需要创建 ticket!
检查兼容矩阵                          
检查 waiting 队列                     

→ 需要 MDL_lock 的读写锁             → 只需要原子操作
</code></pre>
<p><strong>实际的 CAS 循环比"计数器++"复杂得多。</strong> 一次 CAS 同时完成三个检查：</p>
<pre><code class="language-cpp">// MySQL 9.6.0, sql/mdl.cc:2900（简化）
// m_fast_path_state 的位布局：
//   [高位: 3组计数器] [低位: 3个状态位]
//   状态位: HAS_SLOW_PATH | HAS_OBTRUSIVE | IS_DESTROYED

bool try_acquire_lock_impl(MDL_request *req, MDL_ticket **out_ticket) {
    uint64 old_state, new_state;
    do {
        old_state = lock->m_fast_path_state.load();
        
        // 检查 1: MDL_lock 是否已被标记为销毁？
        if (old_state &#x26; IS_DESTROYED) goto slow_path;
        
        // 检查 2: 是否有 obtrusive 锁在 waiting？
        if (old_state &#x26; HAS_OBTRUSIVE) goto slow_path;
        
        // 检查 3: 是否为 unobtrusive 锁类型（SR/SW/S/SH）？
        if (!is_unobtrusive_lock(req->type)) goto slow_path;
        
        // 计算新状态: 对应计数器 +1
        new_state = old_state + unobtrusive_lock_increment;
        
    } while (!lock->m_fast_path_state.compare_exchange_weak(old_state, new_state));
    // CAS 循环：失败就重试。只有当计数器值和状态位都等于 old_state 时才交换
    
    // CAS 成功 → fast path granted!
    return true;
    
slow_path:
    // 走传统路径：加写锁 → 检查兼容矩阵 → 入 granted 或 waiting
}
</code></pre>
<p><strong>三个检查决定了是否走 fast path：</strong></p>
<ol>
<li><code>IS_DESTROYED</code> — MDL_lock 即将被删除（引用计数归零），不能再用</li>
<li><code>HAS_OBTRUSIVE</code> — 有重锁（SU 以上）在 waiting，fast path 先物化</li>
<li>锁类型必须是 SR/SW/S/SH 之一</li>
</ol>
<blockquote>
<p>📌 源码位置：<code>MySQL 9.6.0, sql/mdl.cc:2813-3141</code></p>
</blockquote>
<h3 id="43-fast-path-什么时候退化">4.3 Fast Path 什么时候"退化"？</h3>
<p>当有一个 incompatible 的锁请求进入 waiting 队列时，所有 fast path ticket 必须"物化"（materialize）——补入 granted 链表。这样新来的重锁才能看到"有哪些轻锁挡在我前面"。</p>
<pre><code>事件: ALTER TABLE 请求 SU 锁（与 SR 不兼容）
→ fast path SR ticket 全部物化到 granted 链表
→ ALTER 看到 granted 中的 SR ticket，确定自己必须等待
→ ALTER 进入 waiting 队列
</code></pre>

<h2 id="5-死锁检测">5. 死锁检测</h2>
<h3 id="51-什么时候触发">5.1 什么时候触发？</h3>
<p><strong>每次线程准备进入等待队列时。</strong> 检测发生在 <code>timed_wait()</code> 之前。</p>
<h3 id="52-怎么检测">5.2 怎么检测？</h3>
<pre><code class="language-mermaid">graph TD
    A[当前线程 A 即将等待] --> B[构建等待图&#x3C;br/>节点=MDL_context&#x3C;br/>边=A在等B持有的锁]
    B --> C[从 A 开始 DFS 遍历]
    C --> D{回到已访问节点?}
    D -->|否| E[继续 DFS → 无死锁 ✅]
    D -->|是: 发现环!| F[选出环中权重最低的线程&#x3C;br/>标记为 VICTIM]
    F --> G[通知 VICTIM 线程&#x3C;br/>放弃等待,回滚事务]
</code></pre>
<h3 id="53-死锁检测的-visitor-模式">5.3 死锁检测的 Visitor 模式</h3>
<p>MDL 的死锁检测使用经典的 <strong>Visitor 模式</strong>——<code>Deadlock_detection_visitor</code> 遍历等待图的每个节点：</p>
<pre><code class="language-cpp">// MySQL 9.6.0, sql/mdl.cc:294-410
class Deadlock_detection_visitor {
    static const uint MAX_SEARCH_DEPTH = 32;  // DFS 深度硬上限
    uint m_current_search_depth;
};

// 三步曲：
enter_node(node):    // 进节点 → depth++，超过 32 直接判死锁
inspect_edge(node):  // 查边 → 回到起点 = 发现环!
leave_node(node):    // 出节点 → depth--，回溯时更新牺牲品
</code></pre>
<p><strong>为什么上限是 32？</strong> 防止恶意构造的深度等待链导致栈溢出。超过 32 层直接判定为死锁——这在实际场景中几乎不可能误判（正常的等待链不会超过 5-6 层）。</p>
<h3 id="54-为什么是-while-循环">5.4 为什么是 while 循环？</h3>
<pre><code class="language-cpp">// MySQL 9.6.0, sql/mdl.cc:4049-4089
void MDL_context::find_deadlock() {
    while (true) {
        Deadlock_detection_visitor dvisitor(this);
        if (!visit_subgraph(&#x26;dvisitor)) break;  // 无环 → 退出
        
        victim = dvisitor.get_victim();
        victim->m_wait.set_status(MDL_wait::VICTIM);  // 杀死牺牲品
        
        if (victim == this) break;  // 自己被杀了 → 退出
        // 继续循环：打破一个环后，等图中可能还有其他环！
    }
}
</code></pre>
<p><strong>一次 kill 一个，直到无环为止。</strong> 因为等图中可能有多个不相交的环——杀死一个牺牲品打破环 A，但环 B 可能仍然存在。</p>
<h3 id="55-死锁示例">5.5 死锁示例</h3>
<pre><code>线程 A: 持有 db1.t1 的 SR 锁 → 现在请求 db1.t2 的 X 锁 → WAITING
线程 B: 持有 db1.t2 的 SR 锁 → 现在请求 db1.t1 的 X 锁 → WAITING

等图: A → B (A 等 B 释放 t2 的锁)
      B → A (B 等 A 释放 t1 的锁)
      形成环 → 死锁!

检测结果: 比较 A 和 B 的权重，选权重低的作为 VICTIM
</code></pre>
<h3 id="56-谁会被选为-victim">5.6 谁会被选为 VICTIM？</h3>
<p>权重计算考虑：</p>
<ul>
<li><strong>锁类型越强，权重越高</strong>（持有 X 锁的不容易被牺牲——"他已经走到最后一步了"）</li>
<li><strong>等待时间越长，权重越低</strong>（等得久更容易被牺牲——"你已经等很久了，再等也没意义"）</li>
<li><strong>有 commit order wait 的线程权重更高</strong>（已经在等 binlog flush，再被牺牲会导致级联回滚）</li>
</ul>
<p>选牺牲品策略：遍历环中所有节点，<code>opt_change_victim_to()</code> 比较权重，始终保留权重最低的。最终 <code>set_status(VICTIM)</code> 通知该线程，它自己醒来后回滚事务、释放持有的所有锁。</p>
<hr>
<h3 id="56-piglet-与-hog等待优先级矩阵">5.6 Piglet 与 Hog：等待优先级矩阵</h3>
<p><strong>这不是死锁检测——这是防止饥饿的优先级调度机制。</strong></p>
<p>MDL 有 4 组等待优先级矩阵，用来决定"当有多个 waiting 请求时，谁先被授予"：</p>
<table>
<thead>
<tr>
<th>矩阵</th>
<th>触发条件</th>
<th>效果</th>
</tr>
</thead>
<tbody>
<tr>
<td><strong>正常优先级</strong></td>
<td>默认</td>
<td>按 FIFO 顺序</td>
</tr>
<tr>
<td><strong>Piglet 提升</strong></td>
<td>SW（写锁）waiting 数量 > <code>max_write_lock_count</code></td>
<td>提升 SW 的优先级——防止写入饥饿</td>
</tr>
<tr>
<td><strong>Hog 提升</strong></td>
<td>SU/SNW/SNRW 等待超过阈值</td>
<td>提升重锁优先级——防止 DDL 被无限拖延</td>
</tr>
<tr>
<td><strong>高优先级锁</strong></td>
<td>SH（高优先级共享）</td>
<td>始终优先授予</td>
</tr>
</tbody>
</table>
<pre><code class="language-cpp">// MySQL 9.6.0, sql/mdl.cc:470-534
// waiting_incompatible 矩阵决定了：
// 一个 waiting 中的请求是否可以"插队"到 granted 中另一个请求前面
</code></pre>
<p><strong>实际效果：</strong> 如果 100 个 SR 连续不断（一直有新的读请求），一个 waiting 的 SW 理论上会被永远阻塞。但当 waiting SW 数量超过 <code>max_write_lock_count</code>（默认值），piglet 机制提升 SW 优先级——新来的 SR 不再被授予，直到 SW 被唤醒。</p>
<p>DDL 同理：超过阈值后 hog 机制提升 SU/X 的优先级，防止 DDL 被 DML 洪流饿死。</p>

<p>📌 源码位置：<code>MySQL 9.6.0, sql/mdl.cc:470</code> <code>waiting_incompatible</code> 矩阵</p>
</blockquote>

<h2 id="6-实战select-和-alter-table-的对决">6. 实战：SELECT 和 ALTER TABLE 的对决</h2>
<p>这是最经典的 MDL 交互场景——也是 <code>Waiting for table metadata lock</code> 报错的根源。</p>
<pre><code class="language-mermaid">sequenceDiagram
    participant S as SELECT (线程A)
    participant M as MDL_map
    participant L as MDL_lock(db1.t1)
    participant A as ALTER TABLE (线程B)
    
    Note over S,A: ⏰ T1: SELECT 先开始
    S->>M: 请求 SR on db1.t1
    M->>L: find_or_insert
    L->>S: ✅ granted (fast path)
    Note over S: SELECT 正在读数据...
    
    Note over S,A: ⏰ T2: ALTER TABLE 来了
    A->>M: 请求 SU (upgradable)
    L->>L: SU 和 SR 不兼容!
    L->>A: ⏳ 进入 waiting 队列
    Note over A: ALTER 等待...&#x3C;br/>"Waiting for table metadata lock"
    
    Note over S,A: ⏰ T3: ALTER 需要升级 SU→X
    A->>L: 升级请求: SU → X
    L->>L: X 和 SR 更不兼容. 继续等.
    
    Note over S,A: ⏰ T4: SELECT 终于完成
    S->>L: 释放 SR
    L->>L: reschedule_waiters()
    L->>A: ✅ grant SU
    L->>A: ✅ 升级 SU → X
    Note over A: ALTER 持有 X 锁&#x3C;br/>可以安全修改表结构了
</code></pre>
<p><strong>关键理解：</strong></p>
<ul>
<li>在 T2-T4 之间，<strong>任何新的 SELECT 也能正常获取 SR</strong>（因为 SR 和 waiting 的 SU 兼容）——所以读不阻塞</li>
<li>但 ALTER 必须等<strong>所有已有的 SR 释放</strong>后才能开始</li>
<li>如果有一个长事务（一直不释放 SR），ALTER 就永远等下去</li>
</ul>

<h2 id="8-常见坑与-faq">8. 常见坑与 FAQ</h2>
<h3 id="-坑-1长事务阻塞-ddl">🔴 坑 1：长事务阻塞 DDL</h3>
<pre><code class="language-sql">-- 会话 A: 开启事务后忘了关闭
BEGIN;
SELECT * FROM t1;  -- 获取 t1 的 SR (TRANSACTION 级别)
-- 去吃午饭了...

-- 会话 B:
ALTER TABLE t1 ADD COLUMN c4 INT;  
-- 💥 永远等待！A 不提交，SR 不释放
</code></pre>
<p><strong>预防措施：</strong></p>
<pre><code class="language-sql">-- 设 DDL 等待超时（MySQL 8.0+）
SET SESSION lock_wait_timeout = 60;  -- 60 秒超时

-- 全局设置
SET GLOBAL lock_wait_timeout = 300;
</code></pre>
<h3 id="-坑-2ddl-被-dml-洪流饿死">🔴 坑 2：DDL 被 DML 洪流饿死</h3>
<p>即使没有长事务，如果有持续不断的短事务（每秒 1000 个 SELECT），每个都获取瞬间的 SR，ALTER 可能一直等不到"所有 SR 都释放"的那个瞬间：</p>
<pre><code>时间线:
T1: ALTER 请求 SU → waiting
T2: 100 个新 SELECT 获得 SR → ALTER 还在等
T3: 那 100 个释放 → 但瞬间又有 200 个新 SELECT 获得 SR
T4: 释放 → 新 SELECT → 释放 → 新 SELECT ...
T∞: ALTER 永远等不到"所有 SR 都释放"的那一刻
</code></pre>
<p><strong>这是 Piglet/Hog 机制存在的意义。</strong> 当 waiting 中的重锁数量超过阈值，新来的 SR 不再被直接授予——给 ALTER 一个执行窗口。</p>
<p><strong>额外防护：</strong></p>
<pre><code class="language-sql">-- 在低峰期执行 DDL
-- 或用 pt-online-schema-change（创建影子表 + 触发器 + 分批拷贝）
</code></pre>
<h3 id="-坑-3inplace-ddl-的-x-锁窗口">🔴 坑 3：INPLACE DDL 的 X 锁窗口</h3>
<pre><code class="language-sql">ALTER TABLE t1 ADD COLUMN c4 INT, ALGORITHM=INPLACE;
</code></pre>
<p>三阶段：</p>
<ol>
<li>获取 SU → 准备阶段（DML 继续 ✅）</li>
<li><strong>短暂升级到 X → 提交元数据变更</strong>（所有 DML 阻塞 ❌）</li>
<li>释放 X</li>
</ol>
<p>虽然是"在线"DDL，但第 2 步有一个必须的 X 锁瞬间。高并发下这个瞬间可能导致大量连接堆积。</p>
<h3 id="-坑-4mdl-与-innodb-锁的交互死锁">🔴 坑 4：MDL 与 InnoDB 锁的交互死锁</h3>
<pre><code class="language-sql">-- 会话 A
BEGIN;
UPDATE t1 SET c = 1 WHERE id = 1;  -- 持有 InnoDB 行锁
ALTER TABLE t2 ADD COLUMN c4 INT;  -- 请求 t2 的 SU → 等会话 B 的 t2 SR

-- 会话 B
BEGIN;
SELECT * FROM t2;                   -- 持有 t2 的 SR
UPDATE t1 SET c = 2 WHERE id = 1;   -- 等会话 A 的 InnoDB 行锁 → 阻塞!
</code></pre>
<p>这是一个混合死锁——A 等 B 的 MDL，B 等 A 的 InnoDB 锁。MDL 死锁检测<strong>看不到</strong> InnoDB 层的等待关系，InnoDB 的 <code>lock_deadlock_detect</code> 也看不到 MDL 层的等待关系！</p>
<p><strong>结果：</strong> <code>innodb_lock_wait_timeout</code> 超时（默认 50 秒）后会话 B 报错回滚。MDL 侧的死锁检测器不被触发——因为等图里没有 MDL 层的环。</p>
<p><strong>防范：</strong> 事务中尽量不倒序访问表。先统一获取 MDL（打开表），再做 DML。</p>
<h3 id="-坑-5lock-tables-引发死锁">🔴 坑 5：LOCK TABLES 引发死锁</h3>
<pre><code class="language-sql">-- 会话 A
LOCK TABLES t1 READ;
SELECT * FROM t2;     -- 隐式请求 t2 SR → 等会话 B

-- 会话 B  
LOCK TABLES t2 READ;
SELECT * FROM t1;     -- 隐式请求 t1 SR → 等会话 A
                      -- 💥 死锁!
</code></pre>
<p><strong>建议：</strong> 用 START TRANSACTION + InnoDB 行锁代替 LOCK TABLES。</p>
<hr>
<h3 id="faq">FAQ</h3>
<p><strong>Q: <code>Waiting for table metadata lock</code> 超时时间是多少？</strong></p>
<p>由 <code>lock_wait_timeout</code> 控制，默认 31536000 秒（一年！）。这意味着如果不设超时，DDL 会一直等。<strong>强烈建议在生产环境设一个合理的值。</strong></p>
<p><strong>Q: MDL 和 InnoDB 行锁是什么关系？</strong></p>
<p>两层独立：Server 层 MDL（保护表结构）→ 引擎层 InnoDB 锁（保护行数据）。一条 UPDATE 先拿 MDL SW，成功后再拿 InnoDB 行锁。两者互不可见——MDL 死锁检测只能看到 MDL 等待图。</p>
<p><strong>Q: 如何避免 Online DDL 阻塞业务？</strong></p>
<table>
<thead>
<tr>
<th>策略</th>
<th>说明</th>
</tr>

<tr>
<td>设 <code>lock_wait_timeout</code></td>
<td>DDL 最多等 N 秒，超时自动放弃</td>
</tr>

<td>低峰期执行</td>
<td>减少 DML 并发度，更容易拿到 SU/X</td>

</tr>

<td>事务越短，SR 持有越短，DDL 等到窗口的概率越大</td>
