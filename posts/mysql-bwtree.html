Bw-Tree：面向新硬件平台的 B 树（2013）

Version: MySQL 9.6.0

Bw-Tree（SIGMOD 2013）

面向新硬件平台的 B 树

作者： Justin Levandoski、David Lomet、Sudipta Sengupta（Microsoft Research）

1. 背景

传统 B 树（如 InnoDB 的 B+树）使用闩锁耦合实现并发：读者或写者从根到叶对每一页加闩锁，在子页被安全闩锁之前一直持有父页闩锁。此协议在现代硬件上有两个根本问题：

  1. 根页和近根页成为争用热点——每次访问都从根开始，在高并发下根页闩锁获取甚至串行化了只读流量。
  2. 缓存行争用上的闩锁开销——闩锁使用原子操作（compare-and-swap），迫使缓存行跨核心失效。B 树的分扇出意味着根页被每个事务触碰，即使没有写者活跃也产生每次访问的缓存一致性惩罚。

BwTree 通过使每一个 B 树操作都无闩锁来解决此问题，包括结构修改（分裂/合并）。

2. 问题

传统闩锁耦合 B 树在规模化时的具体问题：

  1. 根闩锁瓶颈：在 32+ 核时，根页闩锁获取串行化所有并发访问。即使是读闩锁（共享）也会导致缓存行弹跳。
  2. 分裂级联：页分裂沿树向上传播。每层需要排他闩锁，在分裂期间阻塞该子树上的所有读者和写者。
  3. NUMA 不友好：单个 NUMA 节点上的闩锁变量产生与核心数成正比的跨插槽一致性流量。
  4. Flash/SCM（存储级内存）不匹配：闩锁假定 DRAM 延迟（约 100ns）。在 SCM 上（约 1-10μs），在 SCM 访问期间持有闩锁会阻塞所有其他线程微秒级别时间。

核心洞见：闩锁保护的是结构修改（分裂/合并），但如果结构可以原子性地改变，读操作根本不需要闩锁。

3. 方案

BwTree 使用三个关键创新来消除闩锁：

3.1 映射表间接层

  • 一个全局映射表（哈希映射）将逻辑页 ID（PID）映射到物理页指针。
  • 每个 B 树节点由 PID 标识，而非直接指针。
  • 读取一个页：在映射表中查找 PID → 获取物理指针 → 读取页内容。
  • **结构修改改变的是映射表，而非页本身。**举例：页分裂创建两个新页（PID_new_left、PID_new_right），在映射表条目上做 CAS：old_PID → [PID_new_left, PID_new_right]。
  • 读者永远不会看到不一致的树，因为映射表变更是原子的（单次 CAS）。

3.2 增量更新（无原地写入）

  • 写入永远不原地修改页。相反，一个增量记录被前插到页的增量链上。
  • 增量记录描述单次修改："插入键 K，值 V"或"删除键 K"。
  • 基础页（原始页内容）一旦创建不可变。
  • 从页 P 读取键 K：从最新到最旧遍历增量链，首次匹配为胜。
  • 增量链是单向链接的。前插增量记录 = 对页的增量链头指针做 CAS。

3.3 基于 CAS 的结构修改（SMO）

  • 页分裂：创建两个新页（左/右）。构建反映分裂边界的增量链。在映射表上做 CAS 重定向旧 PID → [左 PID，右 PID]。
  • 页合并：在映射表上做 CAS 将两个 PID 重定向 → 单个合并后的 PID。
  • 节点合并：当页的增量链太长（可配置阈值），创建一个新的合并页（基础页 + 全部增量应用后的结果），在映射表上做 CAS 指向新页，淘汰旧页。
  • SMO 期间无闩锁：分裂期间执行读操作的线程要么看到旧映射（仍然有效，分裂尚未完成），要么看到新映射（分裂完成）。两者均为一致性视图。

垃圾回收

  • 旧页（被分裂/合并/节点合并所替代）通过基于 epoch 的回收释放（与 Silo 相同的 EBR 概念，但应用于页节点而非事务状态）。
  • Epoch 管理器推进 epoch，线程公布其当前 epoch，当所有线程都推进到超过页的替代 epoch 后释放旧页。

4. 权衡

收益 代价
所有 B 树操作零闩锁争用，包括分裂/合并 增量链遍历成本：从有 100 条增量记录的页中读取键 K 需要遍历 100 个节点——每次读取成本为 O(增量链长度) 而非 O(1)
读者永不阻塞，写者永不阻塞读者 内存开销：映射表（约 8 字节/页）、增量记录（每次修改的头 + 载荷）、等待 GC 的旧页。较传统 B 树增加 20-40% 内存开销
缓存友好：映射表条目上的 CAS 触碰一个缓存行，对比页闩锁触碰该页的缓存行 合并 CPU 成本：增量链变长时，合并重建整个页——是一个延迟指标不可见的 CPU 尖峰
NUMA 友好：映射表可每 NUMA 节点分区 EBR 依赖：与 Silo 相同的 epoch 延迟问题。慢线程阻塞旧页 GC → 内存无限增长
直接在 Hekaton（SQL Server）内部使用——非独立原型 无基于磁盘的版本：BwTree 假设 DRAM（后来适配了 SCM）。映射表必须装入内存。磁盘上的页需要不同的间接层

5. 论文摘要中未提及的失败模式

来自后续工作（OpenBwTree，CMU 2016-2018；Ziqi Wang 等，SIGMOD 2018）：

  1. 增量链长度爆炸：在更新密集型工作负载下，增量链在合并线程介入前常规达到 500+ 记录。点读从 O(log N) 退化到 O(增量链长度)，将 3 页访问的查询变成 500 次指针遍历。OpenBwTree 的修复：主动合并在 8 条增量时触发（激进）或 64 条（平衡）。

  2. 内存压力下的合并停顿：当 GC 跟不上（epoch 线程慢），旧页堆积。合并线程运行在自己的 epoch 中，但如果 GC epoch 滞后，合并产生垃圾的速度比释放更快。OpenBwTree 项目记录了在大量插入工作负载下 2 分钟内 8GB+ 僵尸页的堆积。

  3. 规模化下的映射表争用：原始 BwTree 使用单个全局映射表。在 >32 线程时，映射表上的 CAS 成为新的瓶颈（取代了根闩锁）。修复方案：分区映射表（每 NUMA 节点），但这使分裂/合并复杂化——一个 NUMA 节点上的分裂必须对所有分区可见。

  4. 范围扫描不一致：BwTree 的增量链是从最新到最旧的单向链表。向前遍历的范围扫描必须使用闩锁或容忍不一致的时间点读取。Hekaton 的解决方案：范围扫描使用扫描游标上的单独闩锁（而非 B 树），这为唯一无法完全无闩锁的操作重新引入了闩锁。

  5. SCM（Optane）写入放大：当适配为持久内存时（Wang 等，SIGMOD 2018），增量更新模型导致大量 NVM 写入放大，因为每个增量记录是单独的缓存行写入。传统 B 树中的单行更新 = 一次页写入（含 WAL）。在 BwTree 上 SCM = 增量记录写入 + 最终合并写入 = 2-3 倍放大。

6. MySQL 代码邻近度

若将 BwTree 的无闩锁设计移植到 InnoDB 的 B+树：

MySQL 区域 文件 变更内容
B+树页分裂 storage/innobase/btr/btr0btr.cc: btr_page_split_and_insert() 完全重写。当前分裂：排他闩锁父页 + 子页，重新分布记录，更新父页指针。BwTree 分裂：映射表条目上的 CAS。不需要父页指针。
B+树游标 storage/innobase/btr/btr0cur.cc: btr_cur_search_to_nth_level() 当前：闩锁耦合遍历 根→叶，使用 buf_page_get_gen(RW_S_LATCH/RW_X_LATCH)。BwTree：每层做映射表查找，无闩锁。游标状态 = [当前 PID，增量链内位置]。
缓冲池 / 页访问 storage/innobase/buf/buf0buf.cc: buf_page_get_gen() 当前：对 page_id 做哈希查找 → (fix_count++、闩锁、指针)。BwTree：对 PID 做映射表查找 → (无闩锁指针 + 版本)。无 fix_count，无闩锁。
页结构 storage/innobase/page/page0page.cc、page0cur.cc 当前：固定大小 16KB 页，含记录堆 + 页目录。BwTree：基础页 + 增量链 + 不可变基础。记录访问 = 从最新遍历增量链。
自适应哈希索引（AHI） storage/innobase/btr/btr0sea.cc AHI 就是 MySQL 的映射表！它将 (index_id, key_prefix) → page 做了映射。BwTree 将其一般化：PID → 物理页。AHI 可以作为 BwTree 式映射表的种子——它已经做了无锁读取。

MySQL 特有障碍：

  • InnoDB 的压缩页（page_zip）：对压缩页做增量更新会爆炸——每个增量记录需要自己的压缩上下文，合并需要解压 + 应用 + 重新压缩。净效果：压缩率退化 30-50%。
  • 崩溃恢复：InnoDB 的 redo log 记录页级变更。BwTree 的映射表和增量链创建了根本不同的恢复模型：映射表是恢复目标，而非单个页面。Redo log 需要映射表层级的记录。
  • TRUNCATE TABLE：InnoDB 丢弃所有页并释放表空间。BwTree 的映射表需要基于 epoch 的旧页 GC——TRUNCATE 变成了一个有无限完成时间的异步操作。

7. 产品映射

产品 状态 证据
SQL Server Hekaton 已采纳 BwTree 是 Hekaton 内存优化表的范围索引。随 SQL Server 2014 发布。
SQL Server 实时运营分析 已采纳 SQL Server 2016+ 中用于 HTAP 的列存储 + BwTree 混合方案。
Sled（Rust 嵌入式数据库） 已采纳实验 Sled 实现了类似 BwTree 的结构，含增量链和基于 CAS 的 SMO——深受 BwTree 论文启发。
Aurora 未采纳 Aurora 的存储是日志结构的，非基于 B 树。页闩锁不是瓶颈，因为存储是分离的。
MySQL 上游 / InnoDB 未采纳 InnoDB 的 B+树是闩锁耦合的。无映射表，无增量链，无基于 CAS 的 SMO。
Percona / MariaDB 未采纳 同为 InnoDB B+树架构。

8. 性能论断（附实验条件）

**声称：**BwTree 在多核上匹配或超过传统闩锁耦合 B 树的吞吐量。读密集型工作负载下接近线性扩展到 32+ 核。

实验条件：

  • **硬件：**双插槽 8 核 Intel Xeon E5-2690（16 逻辑核），64 GB RAM
  • **数据集：**合成键值对（8 字节键，8 字节值），1000 万记录
  • **工作负载：**读写混合（80/20、50/50）、均匀和 Zipfian 分布
  • **对比对象：**BerkeleyDB B-tree（闩锁耦合，页级闩锁）、Masstree（无锁 trie+B+树混合）
  • **关键结果：**均匀工作负载上比 BerkeleyDB 吞吐量提升 1.5-2 倍；读密集型上与 Masstree 持平；写密集型上比 Masstree 提升 1.2 倍。

注意事项：

  • 评估在 16 核上，非 32+（按 2013 年标准算偏小）
  • 无 NUMA 评估（双插槽，但工作负载未做 NUMA 分区）
  • BerkeleyDB 对比：BDB 不是最先进的闩锁耦合 B 树——它是通用嵌入式数据库。InnoDB 的 B+树优化更充分
  • 增量链长度参数调至最优（每链 8 条增量）——有突发写入的真实工作负载会超出此值
  • GC 开销排除在吞吐量数字外（实验在干净系统上，无持续写入压力）

9. MySQL 生态启示

  1. **InnoDB 的自适应哈希索引（AHI）是出人意料的 BwTree 先驱。**AHI 已经在做键前缀到页的无锁映射。将 AHI 从哈希索引一般化到完整映射表（PID → 物理页），是 MySQL 通往无闩锁 B 树操作的最自然路径。这不是完全重写——这是对现有机制的扩展。

  2. **增量链模型解决了 InnoDB 不存在的问题。**InnoDB 的页是固定大小（默认 16KB），并通过闩锁保护进行原地更新。增量更新以读放大（遍历增量链）换取闩锁开销。在 InnoDB 基于磁盘的模型中，读放大成本（额外的页访问）将是灾难性的。BwTree 仅在页在内存中时才有意义。

  3. **映射表是挂接到 InnoDB 上最困难的一块。**缓冲池（buf0buf.cc）深深耦合于页 ID 是物理块地址这一事实。增加逻辑到物理的映射层意味着每次 buf_page_get_gen() 调用都需要映射表查找。这是热路径上的约 50-100ns 开销（缓存命中的映射表查找 + 表条目上的 CAS）对比当前约 30ns（对 page_id 做哈希 + 闩锁）。

  4. **对 MySQL 而言，"无闩锁页分裂"与"无闩锁读"是可以分离的。**InnoDB 可以在保持读闩锁的同时采用基于 CAS 的页分裂（映射表方法）。这将消除持有时间最长的排他闩锁（分裂沿树向上传播）而不触碰读路径。这种增量路径比完整的 BwTree 移植更现实。
