f:T4f3a,<h1 id="mysql-9x-开发生态报告闭源开发97-lts-与社区分裂">MySQL 9.x 开发生态报告：闭源开发、9.7 LTS 与社区分裂</h1>
<blockquote>
<p><strong>一句话摘要：</strong> MySQL 在关闭公开开发 5 个月后发布了首个 9.x LTS 版本（9.7.0），向社区版释放了 Hypergraph Optimizer 等 8 项企业功能——但社区信任危机、工程团队缩减、Innovation Release 采纳率仅 ~1% 等问题使生态面临前所未有的分裂风险。</p>
</blockquote>

<pre><code class="language-mermaid">timeline

<p><strong>关键事实：</strong></p>
<ul>
<li><strong>Innovation Release 采纳率仅 ~1%</strong>（Percona PMM 遥测数据），7 个 Innovation 版本（9.0~9.6）被市场集体忽略</li>
<li><strong>Percona 跳过了所有 Innovation Release</strong>，直接从 8.0 跳到 9.7 LTS</li>
<li><strong>8.0 系列于 2026-04-21 正式 EOL</strong>，8.0.46 为最终版，不再有安全补丁</li>
</ul>
<p>这不是正常的版本迭代——这是一次断裂之后的重新出发。</p>

<h3 id="-焦点hypergraph-optimizer">⚡ 焦点：Hypergraph Optimizer</h3>
<p>这是 9.7 LTS 最具技术含量的变革：</p>
<pre><code>传统优化器                         Hypergraph 优化器
─────────────                      ──────────────────
左深树枚举 (left-deep tree)  →     超图枚举 (bushy tree, DPhyp 算法)
多表 JOIN 空间受限            →     更大的 JOIN 排序搜索空间
固定策略选择                  →     基于成本在 nested-loop 与 hash join 间抉择
</code></pre>
<p><strong>使用方式：</strong></p>
<pre><code class="language-sql">SET optimizer_switch='hypergraph_optimizer=on';
</code></pre>
<p><strong>⚠️ 注意：</strong> 默认关闭，hint 支持不完整。建议先在灰度环境测试后再上生产。</p>
<p><strong>本地代码关联：</strong> 在 trunk 最后活跃期中，WL#16719（Hypergraph cost model for hash-join spill-to-disk）和 WL#16658（skip scan access methods）都是为 Hypergraph 铺路的代码——这些已经在 GitHub 仓库中可见，但优化器的最终版本在内部开发。</p>

<h2 id="5-趋势判断分裂格局中的关键信号">5. 趋势判断：分裂格局中的关键信号</h2>
<h3 id="-五个关键判断">🧭 五个关键判断</h3>
<p><strong>1. MySQL 进入了"闭源开发 → 定期发布"模式。</strong></p>
<p>GitHub trunk 5 个月零 commit 不是疏忽——是 Oracle 的<strong>刻意选择</strong>。9.7 LTS 的 Hypergraph Optimizer 和其他企业功能是在内部开发、打包、测试后一次性发布的。这对社区意味着：你不再能在 GitHub 上追踪 MySQL 的日常演进。信任模型从"看代码判断方向"变成"看发布猜方向"。</p>
<p><strong>2. Innovation Release 模型已经失败。</strong></p>
<p>~1% 的采纳率说明市场根本不接受每季度一次的"非 LTS"版本。DBA 不愿意在生产环境跑 Innovation Release，Percona 等关键下游完全跳过。Oracle 需要重新考虑这个模型——要么让 Innovation Release 有足够吸引力（比如放更多新功能），要么放弃它。</p>
<p><strong>3. 社区分裂正在加速。</strong></p>
<p>248 名工程师公开信 + 500+ 人联名是 MySQL 历史上最严重的社区信任危机。这不是"要不要升级"的问题——是有没有<strong>可信的未来路线图</strong>的问题。对于新项目，"选 Percona/MariaDB 还是 MySQL 官方"正在从一个技术问题变成一个政治/信任问题。</p>
<p><strong>4. 9.7 LTS 是一个有用的版本，但不解决根本问题。</strong></p>
<p>Hypergraph Optimizer 进社区版是实质利好。OTEL 可观测性、动态数据遮蔽也是。但这些是<strong>当前的礼物</strong>，不是<strong>未来的保证</strong>。核心问题——开发透明度和社区治理——未被触及。</p>
<p><strong>5. HeatWave/Lakehouse 路线图不透明是最大风险。</strong></p>
<p>MySQL 在向数据湖和 AI 方向投入大量资源，但这些功能的交付模式、定价、GA 时间表完全在 Oracle 的黑箱中。如果你基于"MySQL 将来支持 Lakehouse"做架构决策，你赌的是 Oracle 的善意，而不是公开的路线图。</p>
