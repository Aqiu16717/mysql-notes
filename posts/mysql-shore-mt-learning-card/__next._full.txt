23:Tf8a,<h1 id="一张图看懂shore-mt--为什么测量比去锁更重要">一张图看懂：Shore-MT — 为什么"测量"比"去锁"更重要</h1>
<blockquote>
<p><strong>一句话：</strong> 在被所有无锁论文当作基线之前，Shore-MT 是第一个系统回答"锁到底慢在哪"的项目。MySQL 至今没做过同级别的瓶颈测量。</p>
</blockquote>
<hr>
<h2 id="核心机制">核心机制</h2>
<pre><code class="language-mermaid">graph TD
    TXN[OLTP 事务] --> LM[锁管理器&#x3C;br/>lock table hash + 死锁检测 DFS]
    TXN --> BP[缓冲池&#x3C;br/>page latch acquire/release]
    TXN --> LOG[日志管理器&#x3C;br/>log buffer mutex]
    
    LM --> INST1[⏱️ 测量: 锁获取延迟&#x3C;br/>哈希碰撞率&#x3C;br/>死锁检测图深度]
    BP --> INST2[⏱️ 测量: page latch 等待&#x3C;br/>LRU 链锁竞争&#x3C;br/>buffer eviction 停顿]
    LOG --> INST3[⏱️ 测量: log mutex 竞争&#x3C;br/>log flush I/O 停顿]
    
    INST1 --> REPORT[📊 瓶颈报告&#x3C;br/>每个争用点 = % of 事务时间]
    INST2 --> REPORT
    INST3 --> REPORT
</code></pre>
<p><strong>不是新算法——是测量工具。</strong> 在每个争用点插入 cycle-accurate 时间戳，告诉你 CPU 时间花在哪。</p>

<h2 id="wl11624mysql-的shore-mt-回应">WL#11624：MySQL 的"Shore-MT 回应"</h2>
<p>MySQL 8.0 的 <strong>竞争感知事务调度 (WL#11624)</strong> 是 MySQL 对 Shore-MT 发现的工程回应：</p>
<ul>
<li>Shore-MT 发现：锁竞争导致 CPU 空转</li>
<li>WL#11624 方案：<strong>不消除锁，但让竞争的事务排队等</strong>，减少 spin-wait 浪费</li>
<li>这是"减少锁危害"路线，不是"消灭锁"路线</li>
</ul>

<h2 id="记住这三件事">记住这三件事</h2>
<ol>
<li><strong>Shore-MT 是一次诊断，不是一个产品</strong> — 它的价值在方法论："先测量，再优化"</li>
<li><strong>线程本地池是唯一被所有人接受的优化</strong> — MySQL 的 <code>THD::mem_root</code> 已经是了，差距在共享结构</li>
<li><strong>在现代硬件上重做 Shore-MT 式诊断</strong> — Apple Silicon / AMD EPYC 上 InnoDB 的瓶颈在哪？这可能是一篇可发表的工程论文</li>
</ol>
