<h2 id="1-wl16076--gtid-与复制库重构39-commits">1. WL#16076 — GTID 与复制库重构（39 commits）</h2>
<h3 id="是什么">是什么</h3>
<p>MySQL 内部 C++ 库的彻底重构，构建了以下干净抽象层：</p>
<ul>
<li><strong>GTIDs</strong>（<code>gtids</code> 库）：取代 <code>mysql::gtid::Gtid_set</code> 的新数据结构</li>
<li><strong>UUIDs</strong>（<code>uuids</code> 库）：解析与比较</li>
<li><strong>Sets</strong>（<code>sets</code> 库）：通用集合操作</li>
<li><strong>字符串转换</strong>（<code>strconv</code> 库）：安全的 string↔number 转换</li>
<li><strong>数学</strong>（<code>math</code> 库）：<code>int_log</code>、<code>int_pow</code>、求和函数</li>
<li><strong>容器</strong>（<code>containers</code> 库）：<code>Basic_container_wrapper</code>、<code>map_or_set_assign</code></li>
<li><strong>区间与迭代器</strong>（<code>ranges</code>、<code>iterators</code> 库）</li>
<li><strong>元编程</strong>（<code>meta</code> 库）：C++20 风格 concept 回移植到 C++14</li>
<li><strong>调试</strong>（<code>debugging</code> 库）：<code>MY_SCOPED_TRACE</code>、<code>Object_lifetime_tracker</code>、<code>oom_test</code></li>
</ul>

<p>HeatWave/Lakehouse 在 9.x 中仍是重点投资方向：</p>
<ul>
<li><strong>Parquet nested types</strong>（WL#17152）：将 Parquet 支持从 flat schema 扩展到 STRUCT/ARRAY/MAP — 对真实数据湖 workload 至关重要</li>
<li><strong>文件级数据放置</strong>（WL#17186）：控制 Lakehouse 文件到物理存储的映射关系</li>
<li><strong>加载校验</strong>（WL#17124）：新增 ALTER TABLE 操作用于校验 Lakehouse 数据加载</li>
</ul>
<h3 id="-与-heatwave-整体战略的关联">🔴 与 HeatWave 整体战略的关联</h3>
<p>MySQL 9.x 的 Innovation 轨道是 HeatWave 功能的载体 — 这些无法进入 8.0/8.4 LTS。此处的 Parquet/Lakehouse 是服务器内置功能，与云独占的 HeatWave 服务不同。</p>

<h2 id="4-foreign-key-cascadingwl11249">4. Foreign Key Cascading（WL#11249）</h2>
<pre><code>9e1e77fac10 WL#11249 - Support Foreign Key Cascading Operation in server
</code></pre>
<h3 id="可能带来的能力">可能带来的能力</h3>
<p>FK cascading（ON DELETE CASCADE / ON UPDATE CASCADE）传统上在存储引擎层执行。此 WL 将 cascade 逻辑提升到 SQL 层，从而支持：</p>
<ul>
<li>跨引擎级联（InnoDB → NDB 等）</li>
<li>更好的 cascade 失败错误报告</li>
<li>潜在方向：并行 cascade 执行</li>
</ul>
<h3 id="-风险点">🔴 风险点</h3>
<p>FK cascading 本质上是递归的，可能触发死锁。提升到服务器层新增了一条代码路径，必须与存储引擎自身的 FK 执行逻辑协调。</p>
