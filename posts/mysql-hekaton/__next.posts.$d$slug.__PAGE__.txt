f:T3bd5,<h1 id="hekatonsigmod-2013">Hekaton（SIGMOD 2013）</h1>
<h2 id="sql-server-的内存优化-oltp-引擎">SQL Server 的内存优化 OLTP 引擎</h2>
<p><strong>作者：</strong> Cristian Diaconu、Craig Freedman、Erik Ismert 等（Microsoft）</p>
<hr>
<h2 id="1-背景">1. 背景</h2>
<p>到 2011-2013 年，Microsoft 的 SQL Server 团队面临一个残酷的现实：为 90 年代基于磁盘的 OLTP 所设计的锁管理器和闩锁系统，在现代硬件上消耗了 30-40% 的 CPU 周期。即使使用内存表（DBCC PINTABLE），瓶颈在于<em>架构</em>而非存储介质。</p>
<p>Hekaton 是 Microsoft 的答案：一个从零构建的内存 OLTP 引擎，<strong>没有闩锁、没有锁、没有锁管理器</strong>。与 Silo（研究原型）不同，Hekaton 作为生产功能随 SQL Server 2014 一起发布。</p>
<hr>
<h2 id="2-问题">2. 问题</h2>
<p>SQL Server 传统引擎存在三个架构瓶颈：</p>
<ol>
<li><strong>锁管理器 CPU 开销</strong>：即使有页级和行级锁，锁管理器的哈希表、死锁检测图遍历和锁升级逻辑在 OLTP 负载下仍消耗约 15% 周期。</li>
<li><strong>B 树页闩锁争用</strong>：经典的闩锁耦合协议（闩锁父页 → 闩锁子页 → 释放父页）在根页和近根页上产生级联争用。</li>
<li><strong>解释执行开销</strong>：编译后的 T-SQL 经过查询处理器中的逐语句执行，产生每语句的设置/拆除开销，这对于短 OLTP 事务来说占主导地位。</li>
</ol>
<hr>
<h2 id="3-方案">3. 方案</h2>
<p>Hekaton 的架构建立在四个支柱上，全部为生产环境（而不仅是研究原型）设计：</p>
<h3 id="31-无闩锁索引哈希--范围">3.1 无闩锁索引（哈希 + 范围）</h3>
<ul>
<li><strong>哈希索引</strong>：在桶链表上使用 CAS 实现无锁。每个桶是一个单向链表。插入在链表头使用 CAS；删除使用标记后 CAS。无全局锁表，无每桶闩锁。</li>
<li><strong>范围索引</strong>：<strong>BwTree</strong>——一种无闩锁的 B 树变体（见单独的 BwTree 卡片）。使用映射表间接层，使得结构修改（分裂/合并）通过 CAS 在映射表上执行，而非在页节点上加闩锁。</li>
<li><strong>完全无页闩锁</strong>：读者从不获取闩锁。写者使用 CAS 进行原子状态转换。</li>
</ul>
<h3 id="32-多版本并发控制mvcc">3.2 多版本并发控制（MVCC）</h3>
<ul>
<li>每一行有<strong>开始时间戳</strong>和<strong>结束时间戳</strong>（非事务 ID，而是来自单调递增计数器的全局提交时间戳）。</li>
<li><strong>快照隔离</strong>作为默认隔离级别（可选的序列化验证）。</li>
<li>事务读取其 <strong>[begin, end)</strong> 区间包含事务开始时间的版本。</li>
<li>更新：插入 begin=commit_timestamp 的新版本，设置旧版本的 end=commit_timestamp。</li>
<li><strong>无读锁，无读写冲突</strong>：读者从不阻塞写者，写者从不阻塞读者。</li>
</ul>
<h3 id="33-本地编译存储过程">3.3 本地编译存储过程</h3>
<ul>
<li>Hekaton 使用专用编译器将 T-SQL 存储过程编译为<strong>本地机器码</strong>（C → DLL）。</li>
<li>消除了逐语句解释执行的开销。</li>
<li>编译后的过程通过函数调用直接访问 Hekaton 表，在热路径上完全绕过 SQL Server 查询处理器。</li>
<li>结果：亚毫秒存储过程执行（对比解释执行的 T-SQL 1-10ms）。</li>
</ul>
<h3 id="34-乐观多版本并发控制">3.4 乐观多版本并发控制</h3>
<ul>
<li>提交时的<strong>验证阶段</strong>：检查是否有任何事务修改了本事务读集中的行，且提交时间在本事务的开始时间和提交时间之间。</li>
<li>如果验证通过：分配提交时间戳，使所有版本可见，写入持久化存储。</li>
<li>如果验证失败：中止并重试（应用程序的责任）。</li>
<li><strong>无等待图，无死锁检测</strong>：冲突通过中止重试而非等待来解决。</li>
</ul>

<h2 id="5-论文摘要中未提及的失败模式">5. 论文摘要中未提及的失败模式</h2>
<p><strong>来自生产经验和后续论文：</strong></p>
<ol>
<li>
<p><strong>垃圾回收（GC）尾部延迟</strong>：Hekaton 的版本清理器是一个协作线程。在持续高更新率工作负载下（如单表 100K 更新/秒），每键版本链常规超过 1 万行。版本清理器滞后，此时一个点读需要 1 万次指针遍历而非 1 次。隐含后果：当 GC 跟不上时，Hekaton 的吞吐量退化 40-60%（Microsoft 自有性能团队在 2015 年技术文档中记录）。</p>
</li>
<li>
<p><strong>本地编译冷启动</strong>：本地编译存储过程的首次执行需要 C 编译 + JIT DLL 加载。对于调用 50+ 张表的存储过程，编译可能需要数秒。Microsoft 在 SQL Server 2016 中增加了"预编译全部"选项来缓解。</p>
</li>
<li>
<p><strong>验证中止风暴</strong>：在单行高争用下（如 100 个并发事务更新同一计数器），乐观验证对 90%+ 的事务失败。每次中止消耗 CPU，重试循环产生正反馈死亡螺旋。缓解方案是对热点行做哈希分区（如计数器用 8 个桶行），但这需要 schema 变更。</p>
</li>
<li>
<p><strong>检查点文件碎片化</strong>：Hekaton 的持久化模型写入检查点文件（数据 + 增量）。在大量插入/删除更替下，检查点文件碎片化，恢复时间随碎片数量线性增长。Microsoft 在 SQL Server 2016 中增加了自动合并。</p>
</li>
<li>
<p><strong>DMV（动态管理视图）盲点</strong>：Hekaton 的 <code>sys.dm_db_xtp_*</code> DMV 不能直接暴露版本链长度或 GC 队列深度。DBA 在生产 Hekaton 表无声退化时才付出惨痛代价发现了这一点。SQL Server 2017 通过 <code>sys.dm_db_xtp_gc_cycle_time</code> 及相关 DMV 修复。</p>
</li>
</ol>

<h3 id="mysql-特有障碍">MySQL 特有障碍：</h3>
<ul>
<li>**InnoDB 的 undo log 是崩溃恢复的基础。**Hekaton 的 MVCC 将版本行内存储——无 undo log。移植到 MySQL 需要一个新的存储引擎（通过 handlerton 可插拔）或并行的版本存储系统。</li>
<li>**MySQL 的复制依赖 binlog 位置 + GTID。**Hekaton 的提交时间戳排序不能干净地映射到 binlog 的序列事件流。MySQL 中的 Hekaton 式引擎需要一个按时间戳排序的提交对应的新 binlog 事件类型。</li>
<li>**<code>performance_schema</code> 对每次锁等待进行测量。**消除锁意味着这些测量必须改变——Hekaton 表上的 <code>events_waits_current</code> 锁等待将消失，这破坏了所有查询它的监控工具。</li>
</ul>

<h2 id="8-性能论断附实验条件">8. 性能论断（附实验条件）</h2>
<p>**声称：**OLTP 工作负载上比传统基于磁盘的 SQL Server 吞吐量提升 5-20 倍。亚毫秒存储过程执行。</p>
<p><strong>实验条件：</strong></p>
<ul>
<li>**硬件：**双插槽 12 核 Intel Xeon X5670（24 逻辑核），96 GB RAM</li>
<li>**数据集：**自定义 OLTP 工作负载（非 TPC-C——Hekaton 使用更简单的 schema：orders、order_lines、customers）</li>
<li>**工作负载：**100% 内存，本地编译存储过程</li>
<li>**对比对象：**SQL Server 2012 基于磁盘的引擎（带内存 DBCC PINTABLE 表，有页闩锁但无磁盘 I/O）</li>
<li>**关键结果：**单表插入工作负载 1.1M 事务/秒（对比基于磁盘引擎 50K = 22 倍）。多表关联工作负载：250K TPS（对比 15K = 16.7 倍）。</li>
</ul>
<p><strong>注意事项：</strong></p>
<ul>
<li>"20 倍"声明针对单表插入工作负载——最简单可能的 OLTP 操作</li>
<li>对比对象是有页闩锁的基于磁盘的引擎，而非已做闩锁优化的引擎</li>
<li>非 TPC-C 基准——不同 schema，不同争用模式</li>
<li>本地编译开启（对解释执行的 T-SQL 对比不公平）</li>
<li>Hekaton 的吞吐量在间隔分析查询时会显著退化（论文中未提及）</li>
<li>GC 开销未计入吞吐量数字（在无持续版本更替的静态系统上运行）</li>
</ul>
<hr>
<h2 id="9-mysql-生态启示">9. MySQL 生态启示</h2>
<ol>
<li>
<p>**工业生产部署的故事是最有价值的部分。**Hekaton 是本集群中唯一一个发版到数百万用户的论文。它的失败模式（GC 尾部延迟、验证中止风暴、检查点文件碎片化）是<em>真实的</em>——由生产环境中的 DBA 发现，而非在实验室中。如果 MySQL 未来考虑无闩锁引擎，Hekaton 的生产教训比 Silo 的洁净室基准更有价值。</p>
</li>
<li>
<p>**Hekaton 证明了"无锁的乐观 MVCC"在大规模下可行。**核心洞见——基于提交时间戳的 MVCC + 乐观验证替代了锁管理器——已在数十亿生产事务中得到验证。这是 InnoDB 的锁管理器（<code>lock0lock.cc</code>）原则上可以被替代的最强证据。</p>
</li>
<li>
<p>**本地编译是 MySQL 缺乏的巨大差异化能力。**Hekaton 的编译存储过程消除了 <code>sp_instr</code> 解释执行带来的 CPU 开销。MySQL 的存储过程执行（遍历 <code>sp_head::execute()</code>）是纯解释的，没有编译路径。对于以存储过程为主的 OLTP 工作负载，这是 10-100 倍的延迟差异。MySQL 没有任何关于存储过程编译的 WL。</p>
</li>
<li>
<p>**隔离级别功能面的收缩是 MySQL 真正的采纳障碍。**Hekaton 只支持快照隔离和序列化。MySQL 用户日常使用 <code>READ COMMITTED</code> 和 <code>REPEATABLE READ</code>（默认）。MySQL 中一个抛弃这些隔离级别的 Hekaton 式引擎将面临巨大的向后兼容阻力。</p>
</li>
</ol>
